==Modules==
interface.py (Entry point. Runtime & environment validation and setup)
projects_manager.py (Gather the data for new ini files, forward it for creating a new one or overwriting.)
config_manager.py (Load ini files and/or overwrite them with ProjectsManager()'s help. Also set up local user data folders for migration and validation.)
version_logic.py (Create/update the changelogs for projects)
opener.py (Cross-platform file and folder opening utility)

Planned:
cloud_backup.py (Back up the changes into the users preferred cloud service)

==Information==
* Architectural pattern: Delegated Finite State Machine
Meaning when Interface calls ProjectsManager, the parent state blocks and ProjectsManager takes over the Event Loop until the user returns back to the Main menu, then control is "popped" back up the stack to the caller.
Interface does not know how modules handle their tasks, VersionLogic and ProjectsManager are both handled by their seperate Event Loops. ConfigManager acts as the I/O database for .ini files.

* Navigation is stack based.
Modules always return to the caller.
1. Interface - the base of the stack
2. ProjectsManager - pushed onto the stack when "Projects" is selected
3. VersionLogic - pushed onto the stack once a sepcific project is chosen

* Hierarchical State Machine
Why this approach?
- Debugging - the procedural deterministic flow makes logic easy to follow.
- Low overhead - meaning no event listeners and heavy external dependencies.
- Reliability and longevity - it worked years ago and works years from now.
